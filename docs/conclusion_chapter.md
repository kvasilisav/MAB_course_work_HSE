# Глава 4. Заключение

## 4.1. Итоги исследования

Курсовая работа отвечает на продуктовый вопрос: **когда в онлайн-эксперименте по CTR целесообразно адаптивное назначение трафика (MAB), а когда — фиксированный A/B**, и как не ошибиться при интерпретации логов для решения о выкате (rollout).

По **блоку I** (польза онлайн-эксперимента) на синтетике и в pairwise-постановке «две версии карточки» (E1, E6, E8b, E12) подтверждено: Thompson Sampling и ε-greedy снижают cumulative regret на 14–16 пунктов относительно `fixed_ab` при T = 5000 (bootstrap по seed: 95% ДИ разности regret не включает 0). В сценарии двух карточек TS даёт ~140 кликов больше за 10 000 показов. На contextual synthetic (E8b) LinUCB имеет наименьший **средний** regret (26,6 ± 30,0 против 49,2 ± 38,2 у fixed), но из-за большого разброса bootstrap-ДИ разности с fixed **перекрывает 0** — вывод о строгом превосходстве LinUCB осторожный. Это обосновывает MAB как инструмент **оптимизации во время эксперимента**, но не доказывает статистическую значимость для rollout.

По **блоку II** (статистическая значимость) на Monte Carlo-синтетике (E4, E5, E11, E14): naïve z-критерий на логах Thompson Sampling при null даёт rejection rate 14,5% (цель 5%) и 10% в E11 (100 trials); на чистом `fixed_ab` — 3% (E4, 200 trials). Занижение до 3% при номинальном α = 5% объясняется дискретностью Bernoulli, односторонним тестом CTR_B > CTR_A и конечным числом прогонов — не систематической «переконсервативностью» теста. IPS-weighted inference восстанавливает Type I до 5%; при слабом эффекте (+1 п.п.) мощность IPS — 24%, при +3 п.п. — 78%. Naïve peeking на fixed split поднимает Type I с 3% до 11%; OBF и mSPRT восстанавливают контроль (4% и 2% соответственно), но на bandit-логах sequential **не заменяет** IPS.

По **OPE** (E2, E2b/c, E9): пайплайн IPS/SNIPS replay-OPE с propensity (замороженная candidate, clipping, shuffle per seed) работает на учебной и полной подвыборке OBD; уверенного ранжирования политик не получено даже на 1,37 млн событий. На учебной подвыборке `bts/all` (10k) ESS для `fixed_ab` падает примерно вдвое относительно `random/all` (58 vs 123, табл. 3.13); на масштабе E9b (1M) ESS вырастает до ~6k, но ranking по SNIPS остаётся нестабильным. Pairwise-проекция E13 иллюстрирует методологический мост к A/B двух карточек, но не заменяет отдельный эксперимент (272 события, ~1 клик на руку).

По **применимости** (P1): LinUCB при 80 руках ~23 ms на решение против ~0,05 ms у TS — контекстный бандит требует отдельного инженерного обоснования.

### Научная новизна и вклад

Теоретическое различие optimization vs inference, риски naïve A/B на adaptive-логах и базовые OPE-процедуры **известны** [5, 13, 19, 21, 26]. OBD [26] и обзоры Burtini et al. [13], Xiang et al. [21] не ставили задачу единого воспроизводимого контура «польза онлайн-эксперимента → три типа сбоев inference → конкретная процедура исправления» в постановке CTR с открытым кодом.

**Вклад курсовой** — интегративно-методологический: (1) двухблочный воспроизводимый фреймворк в одном репозитории с тестами; (2) эмпирическая карта границ на синтетике и OBD — где MAB выигрывает по regret, где ломается inference, где OPE нестабилен; (3) количественная сверка IPS, OBF и mSPRT в единой постановке (E11, E14); (4) продуктовая матрица решений (табл. 4.1). Новые bandit-алгоритмы не предложены; вклад — в **систематизации и проверке** известных тезисов для защищаемой продуктовой линии.

## 4.2. Продуктовая рекомендация

**Таблица 4.1 — Ситуация → рекомендация → обоснование (эксперимент)**

| Ситуация | Рекомендация | Обоснование |
|---|---|---|
| Нужен **rollout** (гипотеза B vs A) | Отдельный **fixed A/B** | E4: Type I ≈ 3%, мощность ≈ 85% при T = 20k, +1 п.п. |
| Уже есть **логи бандита**, новый A/B невозможен | **IPS-weighted inference** | E11: Type I 5%; мощность при +1 п.п. — 24% |
| **Мониторинг** fixed A/B по ходу | **OBF** или **mSPRT** [19, 33] | E14: peeking 11% → OBF 4%; не для bandit-логов |
| Много рук, быстрый бинарный feedback | **MAB** (TS, ε-greedy) | E1: regret −14…16 vs fixed; E12: +140 кликов |
| Награда зависит от **контекста** | **LinUCB** при допустимой latency | E8b: точечно 26,6 vs 49,2; bootstrap не исключает 0; P1: ~450× медленнее TS |
| Оценка кандидата **без нового трафика** | **OPE (SNIPS)** + propensity | E2/E9: пайплайн; ranking осторожно |
| Logging policy = **adaptive (BTS)** | Осторожный ranking | E2b/c, E9b: ESS в десятки раз ниже random |
| Мало данных, низкий CTR | Без сильных выводов по ranking | E2, E9: SNIPS-разрыв ~0,001 |

**Таблица 4.2 — Стоимость внедрения (относительная шкала, P1 + E1/E8b)**

| Политика | Regret (E1 / E8b) | Задержка, 80 рук (мкс) | Сложность | Нужен propensity? |
|---|---:|---:|---|---|
| `fixed_ab` | 52,6 / 49,2 | 42 | низкая | для OPE — да |
| `thompson_sampling` | **36,6** / 28,3 | 51 | низкая | да |
| `ucb1` | 49,1 | 3 | низкая | да |
| `linucb` | — / **26,6** | **23 080** | средняя | да (+ контекст) |

*Условная экономика (иллюстрация, E1, T = 5000):* при допущении 100 ₽ условной ценности клика разность regret fixed − TS ≈ 16 кликов → ~1600 ₽ упущенной ценности на один эксперимент; абсолютные рубли — модель, порядок величины — из E1.

**Ответ на центральный product decision:** для **максимизации кликов во время теста** — MAB (преимущественно TS); для **статистически обоснованного выката** — отдельный fixed A/B; для **post-hoc** на bandit-логах — IPS; cumulative CTR бандита **не является** доказательством rollout.

## 4.3. План экспериментов на диплом (примерные направления)

Раздел оформлен по требованию комиссии к курсовой: гипотеза → формулировка → метод → критерии успеха → риски. Это **не обязательство** продолжать именно эту тему в дипломе, а показ того, **как данную работу можно развить** при наличии ресурсов. Ниже — примерные направления углубления двухблочного фреймворка (§3.1); формат включает экспертную оценку, где уместно.

### Направление 1. Doubly Robust OPE vs SNIPS на OBD

| Поле | Содержание |
|---|---|
| **Гипотеза** | DR-оценка [27] может снизить смещение и дисперсию SNIPS на adaptive-логах (BTS) при том же объёме данных |
| **Формулировка** | SNIPS_DR стабильнее между seed на `bts/all`; ESS_DR ≥ ESS_SNIPS |
| **Метод** | Реализация DR; streaming-прогон на BTS-подвыборке OBD; 20 seed |
| **Критерии успеха** | Снижение std(SNIPS) между seed ≥ 20%; документированный ranking stability score |
| **Риски** | Нестабильность DR при редких кликах; mitigation — clipping весов, sensitivity analysis |

### Направление 2. CUPED + fixed A/B vs inference на bandit-логах

| Поле | Содержание |
|---|---|
| **Статус в курсовой** | Базовая реализация и E15 на синтетике выполнены; перенос на OBD — для диплома |
| **Гипотеза** | CUPED [7] на fixed split может повысить мощность A/B; схема «MAB для exploration + отдельный CUPED-A/B для rollout» потенциально предпочтительнее naïve inference на bandit-логах |
| **Формулировка** | При том же T мощность CUPED-fixed ≥ мощности IPS при слабом MDE (+0,5–1 п.п. CTR) |
| **Метод** | Синтетика + симуляция pre-experiment covariate; сравнение с E4/E11 |
| **Критерии успеха** | Мощность CUPED-fixed ≥ 90% при +1 п.п. при T, где IPS ≤ 30% |
| **Риски** | Нужны ковариаты; в OBD признаки ограничены — сначала synthetic |

### Направление 3. Non-stationary CTR и деградация inference

| Поле | Содержание |
|---|---|
| **Гипотеза** | При дрейфе CTR [16, 38] regret-выигрыш MAB может сохраняться, а post-hoc inference — деградировать быстрее, чем у fixed A/B с re-randomization |
| **Формулировка** | В non-stationary synthetic Type I IPS и naïve A/B растут быстрее, чем в stationary E4/E11 |
| **Метод** | Среда с дрейфом CTR; sliding-window IPS; сравнение с E1/E11 |
| **Критерии успеха** | Кривая Type I vs drift rate; рекомендации по re-randomization |
| **Риски** | Калибровка drift; начать с 2-arm |

### Направление 4. Продуктовый пилот (реальные логи или расширенный OBD)

| Поле | Содержание |
|---|---|
| **Гипотеза** | Двухблочный playbook (§3.1) **может** переноситься на кейс «две версии карточки» при достаточном трафике и согласии продукта |
| **Формулировка** | На fixed A/B Type I контролируется; параллельный MAB даёт меньший regret при сопоставимом горизонте |
| **Метод** | Пилот или расширенная репликация на OBD; логирование propensity; отчёт по табл. 4.1 |
| **Экспертная оценка** | Полуструктурированные интервью с 3–5 продакт-менеджерами / аналитиками экспериментов: приемлем ли раздельный контур «MAB для кликов + fixed A/B для rollout»; шкала Лайкерта 1–5 по критериям из табл. 4.1 |
| **Критерии успеха** | Согласованный MDE; положительная экспертная оценка приемлемости (средний балл ≥ 4); воспроизводимый отчёт |
| **Риски** | Нет доступа к внутренним логам [18]; mitigation — углублённый OBD + synthetic anchor + экспертные интервью без production-данных |

### Направление 5. Strict contextual OPE для LinUCB

| Поле | Содержание |
|---|---|
| **Гипотеза** | Strict contextual OPE [6, 28] может подтвердить или опровергнуть выигрыш LinUCB из E8b на OBD (где batch-режим невалиден) |
| **Формулировка** | LinUCB ≥ TS по SNIPS на `random/all` с propensity и контекстом |
| **Метод** | Strict evaluator; subsample с контекстом; оптимизация latency (P1) |
| **Критерии успеха** | SNIPS_LinUCB > SNIPS_TS с p &lt; 0,05 bootstrap по seed; latency &lt; 1 ms после оптимизации |
| **Риски** | Вычислительная стоимость; mitigation — `np.linalg.solve`, sparse context |

*Примечание.* Выбор темы диплома может отличаться от перечисленных направлений; курсовая демонстрирует методологическую базу и типовые гипотезы продолжения в области онлайн-экспериментов и CTR.

## 4.4. Ограничения, переносимость и воспроизводимость

Работа ограничена одним доменом (fashion e-commerce, OBD), одной метрикой (CTR), Monte Carlo inference без production A/B. **DR** [27] описан в обзоре и §4.3, но не реализован. **CUPED** [7] проверен в E15 на синтетике; на OBD не переносился. Non-stationarity не проверялась (направление 3 в §4.3).

**Код и воспроизводимость.** Исходный код, тесты и скрипты экспериментов — в локальном репозитории курсовой; полный прогон — `run_all.ps1` / `run_all.sh` (pytest, основные эксперименты, фигуры). Метод replay-OPE уточнён: candidate-политика заморожена при оценке, propensity клипируется, события перетасовываются per seed (§3.8, приложение А). Сводные CSV для сверки с текстом — в `outputs/` (см. `.gitignore`). Публичная выкладка на GitHub **возможна** при согласовании с требованиями кафедры к открытию учебных работ; на момент сдачи ограничение — отсутствие отдельного публичного зеркала, воспроизведение — по локальной копии и README.

При наличии продуктовой платформы и данных каркас из §3.1 **может** быть адаптирован под пилот (§4.3, направление 4) без изменения логики блоков I и II.
